# AI 幻觉：认知与应对策略

> 日期：2026-03-22
> 场景：自用编码助手 / MemoryPalace RAG 系统

## 幻觉的根源（代码场景）

| 来源 | 表现 |
|------|------|
| 知识截止日期 | 用了过时的 API、已废弃的函数 |
| 凭空编造 | 库名/参数名根本不存在 |
| 上下文不足 | 不知道项目结构，乱猜 |
| 置信度不校准 | 不确定也说得像真的 |

## 核心判断

幻觉的本质是**模型说了无法验证的话**。
解法不是让模型"更谨慎"，而是**让答案可验证**。

## 三个层次的应对

### Level 1：提示层（今天就能做）
- 系统提示加约束：*"不确定时明确说不确定，不要编造函数名或参数"*
- 追问验证：回答后让 Claude 自查 *"这段代码里有没有你不确定的部分？"*

### Level 2：上下文层（RAG）
- 把官方文档/项目文件塞进上下文，让模型基于事实而非记忆回答
- MemoryPalace 的做法：embedding 检索 → top hits → LLM 生成答案
- 关键：检索命中要准，命中差的情况下 LLM 仍会幻觉

### Level 3：验证层（可执行）
- 让 Claude 用 tool use 实际跑代码，而不是凭记忆描述结果
- Mycroft 审查模式：Holmes 给出答案后，Mycroft 独立审查置信度
- 原则：**答案能被执行/查询验证 > 语言模型的自我评估**

## MemoryPalace 里的相关机制

- **Mycroft 审查**：complex 路径第 3 次 LLM 调用，专门审查置信度
- **子问题拆解**：complex 问题拆成 2-3 个子问题分别检索，减少单次检索覆盖不全
- **去重合并 top8**：多路检索结果合并，提高召回率
