# 面向研发团队的 AI 共建知识库架构模板 (脱敏开源版)

这是一套高度提炼、面向“游戏研发组（策划+程序）”或“技术驱动型产研团队”量身定制的 AI 共学共建知识库架构标准。

它抛弃了传统的沉重业务流转框架，**以“AI 提效探索与资产沉淀”为核心**，采用 `探索 -> 沉淀 -> 工程 -> 数据` 的四层极客设计理念。

---

## 🏗️ 核心架构全景 (四层模型)

### 📁 1. 🧪 实验场 (Lab Space)
> **定位**：探索区与“垃圾场”。门槛最低，鼓励试错。
> **适用人群**：全员
- **📄 新工具探险日志** (飞书文档)：记录任何新出的 AI 模型、生图网站的 5 分钟试用体验。
- **📄 在研 Prompt 草稿箱** (飞书文档)：尚未成熟、仍在调教中的提示词、分镜约束词等。
- **📊 踩坑记录表** (多维表格)：记录 AI 幻觉、自动化脚本挂掉的负面案例（往往比成功经验更有价值）。

### 📁 2. ✅ 武器库 (Arsenal)
> **定位**：经过实战检验、可以直接 Ctrl+C 拿走复用的高效红利库。
> **适用人群**：全员查询
- **📊 Prompt 军械库** (多维表格)：收录成熟的咒语，按“策划 / 程序 / 美术 / 测试”打标签过滤。
- **📄 工作流 SOP 手册** (飞书文档)：固化的提效操作指南（例如“如何用 Cursor 快速定位内存泄漏”）。
- **📄 AI 工具推荐榜** (飞书文档)：定期优胜劣汰的团队认证工具白名单。

### 📁 3. ⚙️ 流水线 (Engineering Pipeline)
> **定位**：工程底座。存放所有自动化接入、脚本配置等硬核技术栈。
> **适用人群**：程序、技术策划
- **📄 MCP (模型上下文协议) 接入指南**：指导如何让本地大模型（如 Claude Desktop）安全读取企业内部飞书文档或私有 Git 库。
- **📄 自动化脚本/代码片段库**：存放用于批量处理业务数据、自动同步表格的 Python/Node.js 源码片段。
- **📄 本地大模型环境部署 SOP**：显卡算力规划、Ollama 或 LM Studio 等本地推理后端的统一部署标准。

### 📁 4. 📊 原料仓 (Data & Raw Materials)
> **定位**：单源真实（SSOT），全剧终唯一的**结构化数据接驳口**。
> **适用人群**：AI 自动化脚本（机读为主），人审为辅。
- **📊 游戏本地化 / 跨国翻译词典** (多维表格)：全局统一名词库，AI 多语言生成时的字典防幻觉约束源。
- **📊 核心数值配置大表** (多维表格)：战斗系数、掉落权重。供高阶脚本在此抓取数据推送到游戏配置或供 AI 进行沙盘演算。

## 💡 配置最佳实践 (Best Practices)

1. **飞书节点类型选择**：
   - 知识库的顶级目录，建议直接使用**「飞书文档 (Docx)」**来充当文件夹。
   - 优势：点开文件夹本身的瞬间，能呈现出一篇漂亮的 README 说明（交代这个目录放什么、别放什么），比单纯的空文件夹体验好无数倍。

2. **多维表格列头规范**：
   - 飞书新建的多维表格原生会带上极其干扰排版的三个废列（`Text[主键]`, `Date`, `Attachment`）。
   - **务必**通过 API 或手动方式：征用并改名第一个锁定的 `Text` 栏作为你的业务主键（例如 "词条名"），并强行删除后面的 `Date`/`Attachment`，使数据表紧凑干净。

3. **协同隔离策略（人工搭骨架，AI填血肉）**：
   - 因为企业级安全限制，机器人不宜直接提权掌控整个空间。最佳管理路径是：**人类管理员负责创建上述四个顶层目录，然后将特定的核心生产子节点（比如原料仓里的词典）定向共享（设为可编辑）给 AI 机器人**。
   - 这样既保证了知识库的安全，又能让 AI 在它的“独立工作间”里无限量快速插入、清洗数据。
