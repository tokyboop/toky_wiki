# Self-Review Loop：AI 自我改进循环模式

> 来源：2026-04-03 群讨论整理，结合软件架构横切/纵切概念。

---

## 核心模式

让 AI 执行任务后立刻 review 自己的输出，提出改进，循环直到收敛。

```
执行任务 → 自我 review → 发现问题 → 修改 → 再 review → ... → 收敛
```

Opus 4.6 通常需要 6-8 轮才会说"没有可改进的地方了"。

---

## 关键问题：收敛条件不可信

模型自评的"OK了"不可信——第7轮说OK，第8轮可能又翻车。

**解法（三段拆法）**：

```
前置 gate（checklist 定好切法）
        ↓
执行任务
        ↓
自我 review（最多 N 轮，强制上限）
        ↓
后置 gate（规则校验，不信模型自评）
        ↓
输出结果 + 变更日志
```

---

## 横切 vs 纵切

本质是**黑盒纠偏频率的逻辑差异**。

| 方式 | 每轮做什么 | 优点 | 缺点 |
|------|-----------|------|------|
| **横切** | 同时检查所有维度 | 纠偏频率高 | 每次浅，容易漏深层问题 |
| **纵切** | 每轮只盯一个维度 | 单维度彻底 | 轮次多，跨维度交叉问题容易漏 |

**结论：self-review 的 checklist 应该纵切**——每轮聚焦一个维度，而不是一次全撒。

前置 gate 的作用就是决定"切几刀、按什么顺序切"。

---

## Review Prompt 模板（改良版）

```
你是一个严格的 [任务类型] reviewer。
你刚才输出了以下内容：
---
[结果]
---
本轮只检查以下单一维度：[当前维度]
判断是否有问题（✅ 或 ❌ + 一句理由）。

如果有 ❌，直接输出完整修改版本（不要只说"应该改"）。
如果 ✅，输出 LGTM + 置信度（0-100）。
```

后置规则：置信度 < 80 → 强制再跑一轮。

---

## 纵切维度示例（按任务类型）

**写 SQL**：
1. NULL 处理是否完整
2. 时区/日期边界是否正确
3. 是否会全表扫描
4. 过滤条件是否符合业务定义

**写运营分析报告**：
1. 数字是否准确、有无前后矛盾
2. 结论是否有数据支撑（不是拍脑袋）
3. 建议是否可操作（不是废话）
4. 是否遗漏重要异常

**写 Prompt**：
1. 角色定位是否清晰
2. 输出格式是否约束明确
3. 是否存在歧义表达
4. 收敛条件是否外部可控

---

## 适用场景

- AI 生成报告后的质量提升
- SQL 写完后的自检
- Prompt 设计后的迭代优化
- 任何"一次生成质量不够稳定"的场景
