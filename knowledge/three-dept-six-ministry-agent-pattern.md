# 三省六部制多Agent协同架构

> 来源：卡兹克 2026-04-02
> 适用：多Agent任务分工设计、复杂决策流程、OPC（一人公司）工作流

---

## 核心思想

三省六部制的本质是**强制角色单一视角 + 流程聚合**：每个环节的参与者只站在自己角度往深处思考，通过流程把不同角度的思考汇聚为一个整体决策。

搬到多Agent框架上，这套制度本身就是最好的 prompt 结构——不需要额外设计协作逻辑。

---

## 架构映射

```
三省六部制
├── 中书省  →  提案Agent：起草多个方案，不做判断
├── 门下省  →  审议Agent：专门挑毛病，可一票否决
├── 尚书省  →  执行统筹Agent：综合审议结果做决断，分派任务
└── 六部    →  并行专职Agent（各管一摊）
    ├── 吏部  →  人力/团队规划
    ├── 户部  →  财务/资源测算
    ├── 礼部  →  品牌/传播
    ├── 兵部  →  竞品/市场分析
    ├── 刑部  →  合规/风险
    └── 工部  →  技术/工程实现
```

---

## 关键洞察

**门下省（审议）的价值**
- 起草方的人永远觉得自己的方案好；专职挑毛病的角色能看到盲点
- 好的审议不是为了挑毛病而挑毛病，而是识别核心假设是否成立
- 示例：提案说"用户会主动晒截图"→ 审议追问"凭什么？截图的驱动力是什么"
- 示例：提案做"心流数据仪表盘"→ 审议指出"心流是反分析的，一量化就消失了"

**并行执行的力量**
- 六部同时展开，汇报质量超过串行单人思考
- 每个 Agent 对应一个真实职能部门的思考框架，深度远超泛化 Agent

## 变体：多模型圆桌（用于思考，非执行）

适合**把一个问题想透**，而非出 PRD。

```
圆桌结构（7 Agent）
├── 主持 Agent  →  推进流程，不参与讨论
└── 6 个模型 Agent（可以是不同模型）

五轮流程：
1. 独立分析  →  各自对议题给出观点，互不干扰
2. 交叉质疑  →  每个 Agent 挑其他人的毛病
3. 回应修正  →  承认被打中的漏洞，反驳不成立的质疑
4. 共识收集  →  哪些点全场认了，哪些点仍有分歧
5. 投票 + 最终报告  →  选意见领袖写总结
```

**与三省六部的区别**：圆桌是扁平制衡，适合探索性议题；三省六部是层级流程，适合执行类任务。

---

## 适用场景

| 场景 | 推荐结构 |
|------|---------|
| 产品立项、复杂决策 | 三省六部（流程驱动） |
| 思想探讨、问题拆解 | 多模型圆桌（扁平制衡） |
| 单一方向执行 | 普通 Agent + Skill |

---

## 工程注意事项

- **上下文是瓶颈**：Agent 数量多、轮次长时，上下文工程决定质量上限
- **角色边界要硬**：审议 Agent 必须不能"帮忙优化方案"，否则失去制衡价值
- **六部可以裁剪**：根据任务性质保留相关职能部门，不必凑齐六个
