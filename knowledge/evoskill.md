## 根本问题

Coding Agent 有通用能力但缺乏领域专长。现有方案要么手工写Skill，要么用进化方法优化低层级产物（prompt/代码），跟特定模型和任务强耦合。
需要一个**在更高抽象层自动发现可复用Skill**的框架。

---

## 核心框架：三Agent进化循环

### 三个Agent（关键：职责隔离）

| Agent | 职责 | 不能做什么 |
|---|---|---|
| **Executor** | 执行任务，使用现有Skill | 不能写自己的Skill（防自修改偏差） |
| **Proposer** | 分析失败案例，诊断原因，建议新Skill | 不能执行任务（防确认偏差） |
| **Skill-Builder** | 把建议具象化为Skill文件夹 | 不能评估Skill是否有效（防自我评价偏差） |

**设计洞察**：三者刻意隔离职责，每个Agent只做自己擅长的事，避免认知偏差。这和制作人系统"PM不设计"是同一种思路。

### 进化循环（5步）

```
1. Executor 在训练集上跑任务 → 收集失败案例
2. Proposer 分析失败 → 诊断根因 → 提出Skill建议
3. Skill-Builder 生成Skill文件夹（SKILL.md + 触发元数据 + 辅助脚本）
4. 在验证集上评估新配置
5. Pareto Frontier 保留top-N，淘汰弱者 → 下一轮迭代
```

### Pareto Frontier 选择机制

- 维护一个大小为k的前沿集G
- 每轮从G中轮询选一个parent，诊断其失败，生成候选
- 候选得分超过G中最弱成员才能入选
- 每个配置用git branch追踪，完全可复现
- **关键**：不同轮次发现的Skill互补（各自捕获不同失败模式），合并后效果最强

### Skill文件夹结构

```
.claude/skills/
└── <skill-name>/
    ├── SKILL.md          # 使用说明
    ├── 触发元数据         # 何时激活
    └── helper scripts    # 辅助脚本
```

兼容 Claude Code / Codex / OpenCode 等支持Skill文件夹的Agent框架。

## 实验结果

| 基准 | 基线 | EvoSkill | 提升 |
|---|---|---|---|
| OfficeQA（美国财政数据推理） | 60.6% | 67.9% | +7.3% |
| SealQA（噪声检索QA） | 26.6% | 38.7% | +12.1% |
| BrowseComp（零样本迁移） | — | — | +5.3% |

**关键发现**：
- skill-merge（合并多轮独立发现的Skill）效果最强，说明不同失败模式产生互补Skill
- SealQA上学到的Skill**零样本迁移**到BrowseComp有效，证明Skill级优化产生可迁移能力
- 用小训练子集就够，不是靠数据量暴力提升

### 两种进化模式

- **skill_only** — 只发现新Skill（推荐）
- **prompt_only** — 只优化系统提示词

## 和我的关系

### 直接可用的洞察

1. **职责隔离防偏差** — EvoSkill的三Agent隔离和我系统的"PM不设计"是同一原理。执行者不能自评，提议者不能自验
2. **失败驱动改进** — 不是随机变异，而是分析"哪里错了"→"需要什么能力"→"生成能力"。比盲目迭代高效得多
3. **Skill合并>单Skill** — 多轮独立进化的Skill互补，合并后最强。启示：不要只保留"最好的一个"，而是保留"互补的一组"
4. **Skill可迁移** — 在一个任务上学到的Skill在另一个任务上零样本有效，说明Skill级抽象层次正确

### 可能的应用

- 给我的Agent系统加自动Skill发现：让Agent在实际项目中失败→分析→自动生成新Skill
- 对knowledge-curator的启发：外部知识消化也可以用"失败驱动"思路——当检索失败时，反推需要补充什么知识
- Pareto Frontier思路可用于管理设计方案：维护一组互补方案而非单一最优

### 需要验证的问题

- Skill文件夹格式是否和我的OpenCode Skill结构兼容？
- 三Agent架构在非编码任务（如设计评审）上是否同样有效？
- 小规模项目是否值得搭建进化循环？
