# 游戏本地化翻译避坑与复盘说明 (Post-mortem)

在处理 `game.csv` 游戏文本本地化的过程中，我们经历了一系列经典的“AI翻译生产事故”以及数据整合问题。为确保以后（无论是本项目还是其它语种/新赛季文本扩充）能避开同类陷阱，特此记录核心踩坑点及解决策略。

## 坑位 1：大语言模型上下文过载导致的“发疯发散”
*   **现象**：由于单次对话或单批处理的文件过长，大模型在最后几条文本（如 `Message_Tutorial_...` 系列弹窗介绍）突然出现严重的语句重复、胡言乱语。例如出现了高达上百字的语义不明废话（如：“*你也必须需且还急切需要完成去硬拼去做到并填满满足另外的一干附带来的前提一连串的重重艰难先决苛刻...*”）。
*   **原因**：当单次输出长度过高或 Prompt 负荷过重时，注意力机制衰退导致的典型“复读机”与强行凑字数现象。
*   **解决方案**：
    1.  **强行分批 (Chunking)**：拒绝超过 400 行以上的单次翻译大单，改用分批切片形式每次处理约 150-200 行文本。
    2.  **二次审计拦截**：写 Python 脚本拦截长度荒谬的字符串。例如设定 `if len(zh_text) > 200` 时触发警告进行人工校验或重新生成。

## 坑位 2：未转义反斜杠与引号破坏 JSON 结构 (JSON Decode Error)
*   **现象**：在将翻译文本包裹在 JSON 代码块中输出时，因为游戏剧情文本常常带有对话自带的引号（如 `"这是一个例子"` 或 `”某某“`），这与 JSON 对象标准的字符串双引号发生冲突，导致应用到 CSV 时的 `json.loads()` 全盘崩溃，抛出 `Expecting ',' delimiter`。
*   **原因**：翻译输出中没能正确转义（Escape）内部英文或者中文的直角双引号（`"`）。
*   **解决方案**：
    1.  如果出现报错，编写定制化 Python Regex 脚本（`fix_json_quotes.py`）只匹配首尾双引号结构，把值中间非法的 `"` 手工替换为 `\"` 再次重构 JSON。
    2.  后期要求大模型在输出游戏文案类型的 JSON 时，必须注意内部引号的双转义或改用中文单反引号/中文引号。

## 坑位 3：富文本标签（Placeholder）的语言缺失
*   **现象**：游戏内的特殊占位符比如 `{TAP}`、`{SEASON PASS}`、`{BUILD ZONE}` 原封不动保留了英文格式。虽然这在单纯的“数字占位符”（如 `{0}` 或 `{1}`）上没问题，但带有英文单词的其实是引擎内用于渲染带颜色文本与特定 UI 图标的富文本锚点。不翻译会导致游戏 UI 指示里出现突兀的英语单词甚至断词。
*   **原因**：缺乏提取与规范；模型有时会忽略大写变量内部的翻译，有时又会把它瞎翻破坏原本大括号格式。
*   **解决方案**：
    1.  提取出所有形如 `{WORD}` 的占位符进行单独建档（构建 `PLACEHOLDER_MAP` 字典），例如 `{TAP}` -> `{点击}`。
    2.  写正则脚本 `re.findall(r'\{[A-Z\s]+\}', string)` 查找，在合并写入 CSV 前做最后一步针对性的无损替换。

## 坑位 4：编码血泪史 (BOM 与 Windows Console 乱码)
*   **现象**：在 Windows 命令行下 `print(...)` 提取含有的中文字符时经常抛出 `gbk codec can't encode character...` 或者中文全部变为问号与乱码。直接重写入 CSV 时也可能因为编码选择错误引入错乱。
*   **原因**：CSV文件原始带 `utf-8-sig` (也就是含有 BOM 头) 的格式编码，但 Windows 的 powershell console 使用的字符集存在转码局限性。
*   **解决方案**：
    1.  所有的读取操作严格使用：`open(file, 'r', encoding='utf-8-sig')` 以剥离 BOM，不受系统默认地区编码影响。
    2.  所有的写入操作同样需要 `encoding='utf-8-sig'` 输出供 Excel 或引擎无缝读取，千万不要用系统默认读写流。
    3.  排查具体文案问题时，不使用 `print()`，而是写出到 `txt` 文件后手工查看，避免 `stdout` 中断脚本执行。

## 坑位 5：空白/占位符的防御性处理
*   **现象**：CSV 中有多条诸如 `TutorialPopup_10~27` 的条目原本就是只有一个特殊记号 `"` ，代表游戏引擎这几个弹窗预定为空。如果盲目让 AI 去翻译，会自行脑补创造没有的原意。
*   **解决方案**：合并翻译前，统计脚本必须先排除“英文原文本身为空或只含有空白字符及标点”的条目，只有携带有效长度英文文本的 `en_idx` 才送入大模型。保留原始 CSV 不动。

***
### 结语
上述问题通过自动化的预处理/后处理 Python 脚本和数据校验流程已经得到完全修正，目前的翻译 Pipeline 已经形成非常稳定的闭环： `抽空缺 -> JSON分批翻译 -> 捕捉并修正 JSON Quote -> 更新并翻译富文本 PlaceHolders -> 覆盖应用回 CSV (With UTF-8-sig)`。之后再有文本补充，可依照此方案作为标准 SOP 处理。
