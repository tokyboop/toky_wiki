## 关键决策 & Insight

### 2026-03-21：活动 log 的现实

- **现状**：没有活动级别 log 表，用道具流水变动间接衡量活动参与度
- **别人家**：研发在活动框架层面就做了 activity_id → 运营直接查。这是框架设计问题，不是运营能后补的
- **痛点**：策划不喜欢加表，老活动推不动，新活动之前也没加 log
- **结论**：不硬磕老活动，新活动有机会再推。道具流水方案够用就先用着
- **方向**：先专注 2-4（自己能控制的），不要漂移重点

### 2026-03-21：异常监控告警方案设计

#### 核心思路

一套 SQL 模板 × 5 种对比维度，定时跑，超阈值推飞书。

#### 5 种对比维度

| 维度 | 基线 | 适用场景 |
|------|------|---------|
| 日环比 | vs 前一天 | 日常波动，最基础 |
| 周同比 | vs 上周同一天 | 排除周几效应 |
| 月整体 | vs 上月同期均值 | 趋势性变化 |
| 活动期对比 | vs 上一次同类活动 | 活动效果好坏 |
| 历史活动对比 | vs 其他活动同期 | 跨活动横向比 |

#### SQL 模板设计

核心模式：**当前值 vs 基线值 → 变化率 → 超阈值则告警**

```sql
-- 通用对比模板（伪代码，需适配实际表名和字段）
WITH current_period AS (
  SELECT {metric} AS val
  FROM {table}
  WHERE {time_col} BETWEEN {current_start} AND {current_end}
  {extra_filter}
),
baseline_period AS (
  SELECT {metric} AS val
  FROM {table}
  WHERE {time_col} BETWEEN {baseline_start} AND {baseline_end}
  {extra_filter}
)
SELECT
  c.val AS current_val,
  b.val AS baseline_val,
  ROUND((c.val - b.val) / NULLIF(b.val, 0) * 100, 1) AS change_pct
FROM current_period c, baseline_period b
HAVING ABS(change_pct) > {threshold_pct}
```

变量化的部分：
- `{metric}`：SUM(amount) / COUNT(DISTINCT uid) / COUNT(*) 等
- `{table}`：道具流水表 / 登录表 / 充值表
- `{time_col}`：日期字段
- `{current_start/end}`、`{baseline_start/end}`：由对比维度决定
- `{threshold_pct}`：告警阈值，比如 30%

#### 监控指标（初版建议）

先从最痛的开始，不贪多：

| 指标 | 数据来源 | 阈值建议 | 优先级 |
|------|---------|---------|--------|
| 日活 DAU | 登录表 | ±15% | P0 |
| 道具消耗量 | 道具流水表 | ±30% | P0 |
| 充值金额 | 充值表 | ±20% | P0 |
| 新增用户数 | 注册表 | ±25% | P1 |

#### 告警通道：飞书机器人

```
飞书群 → 添加自定义机器人 → 拿到 webhook URL
↓
SQL 跑出异常 → 格式化消息 → POST webhook
```

消息模板：
```
📊 数据异常告警
指标：日活 DAU
当前值：12,345
基线值：15,678（昨日）
变化：-21.3% ⬇️
时间：2026-03-21
```

#### 执行管道（两种路径选一）

**路径 A：轻量脚本**（推荐先用）
- Python 脚本，连数据库跑 SQL，结果推飞书
- cron 定时执行（每天 10:00 跑前一天数据）
- 优点：快、可控、不依赖额外系统

**路径 B：BI 工具内建**
- 如果已有 BI 平台（Grafana / Metabase / 自建看板），用它的告警功能
- 优点：可视化 + 告警一体
- 缺点：灵活度受限

#### 下一步

1. **确认表结构**：你的道具流水表、登录表实际叫什么、关键字段是什么
2. **先跑一个指标**：选 DAU 或道具消耗，写出第一条真实 SQL
3. **飞书 webhook 先建好**：群里加个机器人，拿到 URL
4. **Python 脚本串起来**：连库 → 跑 SQL → 判断阈值 → 推飞书
