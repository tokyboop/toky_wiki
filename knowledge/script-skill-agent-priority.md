# 脚本→Skill→Agent 优先级决策框架

> 来源：数字生命卡兹克公众号（2026-03-17）
> 适用：日常工作流设计、AI工具选型、避免过度工程

---

## 核心原则

**能用更简单的工具解决，就不要动更复杂的。**

很多人一听 Agent 厉害，就把所有事情往 Agent 上堆。结果：慢、贵、不稳定。

---

## 三层金字塔

### 第一层：脚本（优先）

**适用场景**：逻辑固定，输入输出确定，不需要判断，不会出现意外情况。

**判断标准**：你能把每一步写死吗？能，就用脚本。

**例子**：飞书自动化、定时数据拉取、格式转换、批量重命名。

**为什么不用 Agent**：脚本能做的事用 Agent 是大炮打蚊子，引入了不必要的不确定性。

---

### 第二层：Skill（次之）

**适用场景**：纯脚本搞不定，需要大模型的泛化能力，但不需要 Agent 的自主决策和多步骤规划。

**判断标准**：逻辑无法写死，但任务是单一封装的能力单元。

**例子**：资讯打分（判断重要性/相关性/时效性）、内容风格检查、文案审查。

**为什么不用 Agent**：Skill 就够了，Agent 是杀鸡用牛刀。

---

### 第三层：Agent（最后）

**适用场景**：无法提前规划好所有步骤，需要根据过程中的具体情况动态调整策略。

**判断标准**：你知道想要什么结果，但不知道中间要经过哪些步骤。

**例子**：开发一个脚本（需要动态规划）、做竞品体验报告（需要根据发现调整方向）。

---

## 最佳运转方式

```
Agent 创造工具
    ↓
工具执行任务
    ↓
不断往下沉淀能力（Agent → Skill → 脚本）
    ↓
Agent 永远只处理那些还没法被固化的部分
```

这个循环一旦跑起来，才是 AI 在真实工作流里落地的样子。

---

## 决策速查

| 问题 | 答案 → 用什么 |
|------|-------------|
| 逻辑能完全写死吗？ | 能 → 脚本 |
| 需要泛化判断但任务单一？ | 是 → Skill |
| 步骤无法预先规划？ | 是 → Agent |
| 不确定用哪个？ | 从脚本开始往上试 |

---

## 常见错误

- 一上来就用 Agent，导致慢、贵、不稳定
- 把固定逻辑的任务做成 Skill，浪费模型调用
- 没有往下沉淀的意识，Agent 一直在做重复性工作
