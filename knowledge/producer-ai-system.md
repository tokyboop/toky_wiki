# 制作人AI辅助系统 — 架构蒸馏

> 蒸馏日期：2026-03-21
> 置信度：C1（自建系统，经验证）

---

## 根本问题

制作人的设计思维（意图树、广深全、理性优先）只存在于脑中，无法规模化复用。
需要一个**内化了制作人思维方式的专业AI系统**，而不是通用助手。

---

## 核心框架：10 Agent + 5 Skill + 4 层记忆

### 三层 Agent 架构

| 层 | Agent | 职责边界 |
|---|---|---|
| **设计层**（想清楚） | intent-architect, design-explorer, design-evaluator, challenger | 意图树→穷举→量化评价→质疑 |
| **知识层**（知道够多） | doc-reader, knowledge-curator | 文档消化、外部知识+置信度管理 |
| **管理层**（拆得对） | pm | 四维度递归拆分（难度/解耦/可验证/聚类），不做设计 |
| **执行层**（做得好） | coder, debug, unity-dev | 项目感知编码、假设-验证调试、Unity闭环 |

**关键约束**：PM 不设计，Coder 写前必扫描项目，所有代码必验证。

### 5 个 Skill（领域知识模块）

system-design / project-initiation / content-design / engineering / numerical-design

### 全流程管线

```
设计（意图树→探索→评价→质疑）
  → PM拆分（四维度递归）
  → 设计细化（循环至可编码）
  → 🧑用户审核拆分方案
  → 并行执行（用例先行→编码→自动测试+Debug→集成）
  → 整体验证
  → 🧑用户审核交付
```

### 人机协作原则

- **开始前**：制作人审核拆分方案，设迭代上下限
- **执行中**：全自动，超限或重大问题才请求介入（不阻断其他子单元）
- **结束后**：制作人审核产出+疑难
- 任意节点可介入或跳过

## 技术栈

| 层 | 选型 | 理由 |
|---|---|---|
| AI运行时 | OpenCode + Claude | 主Agent环境 |
| 浏览器自动化 | Playwright MCP | Web原型可视化验证闭环 |
| 游戏引擎 | Unity MCP | 引擎操控 |
| Web开发 | 原生HTML/JS/Canvas | 零依赖轻量原型 |

### Web-First 策略

```
Web原型验证（快，AI全闭环）→ 确认后迁移Unity
```

AI写码→启动server→Playwright截图→DOM/JS断言→不对就改→秒级迭代。
核心价值：AI能直接看到渲染结果，不依赖Unity编译等待。

---

## 强制规则（6条）

1. **文档优先** — 新工具必须先读文档，没文档停止汇报
2. **置信度标注** — 引用外部知识必标C1-C5
3. **意图树驱动** — 设计先建意图树含优先级分析
4. **验证闭环** — 写完必须验证
5. **项目感知** — 写码前必扫描项目结构
6. **PM不设计** — 设计问题交回设计Agent

## 和我的关系

这是我自己的AI工作流架构。核心洞察：
- Agent按"想/知/管/做"分层，各守边界
- 设计必须经过意图树→穷举→量化→质疑四步，防止拍脑袋
- PM只管拆分调度，保持设计与执行解耦
- Web-First让AI获得视觉验证能力，大幅加速迭代
