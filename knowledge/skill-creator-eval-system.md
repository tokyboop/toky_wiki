## 四大新能力

| 能力 | 作用 |
|------|------|
| **评估系统** | 跑完直接告诉你 skill 好不好用 |
| **基准测试** | 量化通过率、耗时、token 用量 |
| **多代理并行测试** | 每个测试独立干净环境，支持 A/B 盲评，结果不互相污染 |
| **描述调优** | 自动优化 skill 描述，减少触发冲突 |

## 触发描述优化流程

解决多个 skill 触发打架的问题。

```
1. 自动生成测试样本
   ├── 应触发的查询 × 10
   └── 不应触发的查询 × 10（故意加边界情况）

2. 弹出 UI 让你逐条确认
   └── 判断不对的直接关掉

3. 后台优化循环（最多 5 轮迭代）
   ├── 60% 训练集迭代优化
   └── 40% 测试集选最优（防过拟合）

4. 最优描述自动写回 SKILL.md
```

## Skill 执行评估流程

触发准确了，还要测实际任务效果。

```
1. 读取 skill 文件，理解核心流程
2. 问你：测哪个方面（全面/某模块）
3. 自动设计测试场景 + 量化验收标准
4. 启动 4 个并行子代理独立跑测试
   └── 干净环境，独立 token 计数，零上下文污染
5. 浏览器弹出评估页面
   ├── 输出标签页：每个用例结果 + 反馈标注框
   └── 基准测试标签页：有 skill vs 无 skill 对比
```

**反馈会被存储**：下次用 skill-creator 改进时直接带入，形成"测试→发现→修→再测"完整循环。

## 两种 Skill 类型的评估方向

**能力提升型**（教 Claude 做它不擅长的事）
- 测试重点：模型更新后 skill 是否还有存在必要
- 方法：A/B 对比，有 skill vs 无 skill
- 若结果差不多 → skill 可以退休了

**编码偏好型 / Workflow**（让 Claude 按你的流程走）
- 测试重点：有没有老实按流程走，有没有漏步骤、自作主张、遗忘特殊要求
- 不测"结果好不好"，测"流程合不合规"

---

## 关键洞察

**为什么并行测试重要**：串行测试时，前一个任务的上下文会污染后一个结果，你以为是 skill 的功劳，实际是对话历史帮了忙。并行干净环境才能得到真实数据。

**评估是引领方向的**：PDF skill 处理表格出错，通过评估发现是坐标字段缺失导致，针对性修复后问题解决。"自我感觉良好"不如"数据摆出来一眼见真章"。
